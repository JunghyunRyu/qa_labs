# AI에게 git을 맡겨두고 살다가 얻은 후회들  
_“언젠가는 기본기를 해야 한다는 걸 알면서도, 오늘도 또 AI에게 물어본다.”_

## 1. 나는 왜 아직도 git을 검색해서 쓸까

요즘 개발자들은 솔직히 인정해야 한다.  
**기본적인 git 명령조차 매번 검색해서 쓰고 있다.** 나도 그중 하나다.

- `git checkout -b`를 몇 번이나 썼는지 기억도 안 나지만,  
  여전히 정확한 옵션은 AI에게 물어본다.
- 머지 충돌이 날 때마다, 구글 대신 이제는 AI에게 “이거 어떻게 풀어요?”라고 묻는다.
- 어떤 날은 `git reset`이랑 `git revert`를 헷갈려서,  
  결국 “그냥 전체 상황 설명하고, 이제 뭘 치면 되는지 알려줘”라고 부탁한다.

그리고 그 결과, 편해지긴 했다.  
문제를 만날 때마다 **친절한 24시간 git 코치**가 바로 옆에서 알려주는 느낌이다.

하지만 시간이 지나고 보니, 이 편리함 뒤에는 꽤 큰 후회가 남아 있다.

---

## 2. git을 “AI에게 아웃소싱”한 대가

### 2.1 명령어는 실행했는데, 무슨 일이 일어난 건지 모를 때

AI가 알려준 대로 명령어를 순서대로 치면,  
확실히 **일단 문제는 해결된다.**

- 충돌이 사라지고
- `git status`가 깨끗하게 나오고
- 리모트 브랜치도 잘 정리된다

그런데 정작 중요한 건 그 다음이다.

> “지금 방금 내가 한 작업이 **어떤 개념으로 설명되는지** 모른다.”

- 방금 한 게 “히스토리 재작성(rewrite)”인지
- 단순히 “새 커밋 하나 추가”인 건지
- “로컬만 바꾼 건지” 아니면 “원격도 같이 밀어버린 건지”

머리 속에 **모델이 안 서 있으니까**,  
비슷한 상황이 다음에 또 오면 결국 똑같이 AI에게 전부 설명하고 다시 물어본다.

### 2.2 코드가 “증발”하는 순간

한 번쯤은 이런 경험이 있었을 것이다. 나도 그랬다.

- 작업을 한참 했다.
- 머지/리베이스/리셋을 몇 번 반복했다.
- 뭔가 꼬여서, AI가 알려준 대로 이것저것 명령을 쳤다.
- 그리고 어느 순간,  
  **“있어야 할 코드가 안 보인다.”**

분명히 몇 시간 동안 작성한 코드인데:

- 워킹 디렉터리에 없어지고
- `git log`에도 안 보이고
- `git reflog`를 볼 생각조차 못 하고

그냥 **“나 방금 뭘 한 거지…?”**라는 공허함만 남는다.

기술적으로 말하면, 이건 단순히 git 히스토리와 브랜치 개념을 제대로 이해하지 못한 상태에서  
`reset`, `checkout`, `push --force` 같은 명령을 **“설명만 듣고 그대로 따른 결과”**다.

문제는,
그 명령어가 **“왜 필요한지”**와  
**“잘못 쓰면 어떤 일이 벌어지는지”**를 충분히 소화하지 못한 채 썼다는 거다.

그게 결국 **코드가 엎어지는 사건**으로 돌아온다.

## 3. EC2와 보안, 그리고 “비밀번호 안 치는 삶”에 대한 집착

git만 그런 게 아니다.  
AWS EC2를 다루면서도 비슷한 패턴이 반복된다.

### 3.1 git push 때마다 비밀번호를 치는 나

EC2에서 작업하다 보면, `git push`를 할 때마다 아래 같은 프롬프트가 튀어나온다.

- GitHub 사용자 이름
- GitHub 비밀번호 또는 토큰

분명히 어디선가 봤다.

- `ssh-keygen`으로 키 생성해서 GitHub에 등록하면 비밀번호 없이 push할 수 있다는 걸.
- `git config credential.helper store` 같은 걸 쓰면 캐싱이 된다는 걸.

하지만 막상 필요할 때는 **정확한 명령어가 기억나지 않는다.**

그래서 또:

> “EC2에서 계속 GitHub 비밀번호 물어보지 않게 하는 법 알려줘”

라고 AI에게 묻는다.  
그러면 AI는 친절하게 순서를 알려준다.

1. SSH 키 생성
2. GitHub에 공개키 등록
3. 원격 URL을 `https`에서 `ssh`로 변경
4. 필요 시 credential helper 설정

그럼 나는 그대로 따라 한다.  
그리고 며칠 뒤, 새로운 서버를 또 만들게 되면  
**똑같은 질문을 다시 한다.**

결국 **머리에는 남아 있지 않고**,  
매번 “설정은 AI가 알아서 해주는 것”처럼 흘려보내고 있는 거다.

### 3.2 EC2 보안 설정도 “몸에 안 붙는” 느낌

EC2 보안 그룹, UFW, SSM, SSH 키…

- 어떤 포트를 열어야 하는지
- 어떤 IP에서만 열어야 하는지
- SSH 말고 SSM을 써야 안전하다는 것
- UFW를 건드릴 때는 항상 세션을 두 개 열어야 한다는 것

이 모든 걸 **머리로는 이해**한다.  
하지만 시간이 지나면 디테일이 흐려지고,  
정확한 명령어와 옵션은 또다시 AI의 검색창 속으로 들어간다.

그러다 보니, 어느 순간 이렇게 생각하게 된다.

> “왜 나는 이런 기본적인 걸 아직도 외우지 못할까?”

그리고 그 질문에는  
조금의 후회와, 조금의 부끄러움이 섞여 있다.

## 4. 나에게 드는 실망과 후회들

### 4.1 “나, 너무 AI에 기대고 있는 거 아니야?”

AI는 정말 강력하다.  
하지만 어느 순간부터 나는 이런 패턴을 반복하고 있었다.

1. 문제 상황이 생긴다.
2. 내 머리로 구조를 정리하기보다, 바로 AI에게 상황을 길게 설명한다.
3. AI가 던져 준 명령어를 순서대로 입력한다.
4. 문제가 해결된다.
5. **“왜 이렇게 되는지”는 깊게 생각하지 않는다.**

그 결과:

- git은 여전히 **블랙박스**처럼 느껴지고
- 보안 설정은 여전히 **“찾아보면 나오는 것”**으로만 남아 있다.

즉, **지식은 내 것이 되지 못하고**,  
한 번 쓰고 사라지는 “소모품”처럼만 사용된다.

### 4.2 “기본기”를 회피한 대가

반복적으로 느끼는 아쉬움은 이거다.

> “사실 그렇게 어렵지 않은 내용인데,  
>  한 번 맘먹고 정리하지 않아서 지금도 계속 고생하고 있다.”

- `branch`, `commit`, `merge`, `rebase`, `reset`만 정확히 이해해도  
  git은 훨씬 덜 무서울 텐데.
- SSH 키와 credential helper의 개념만 확실히 잡아도  
  GitHub 비밀번호 문제는 영원히 해결될 텐데.
- EC2 보안 그룹, UFW, SSM 구조를 한 번 그림으로 그려 놓기만 했어도  
  “지금 이게 어디에서 막힌 건지” 헷갈리진 않을 텐데.

그걸 알면서도,  
**당장 눈앞의 문제 해결**에만 집중하고 넘어간 날들이 쌓여 왔다.

그래서 지금의 후회는 단순히 “명령어를 외우지 못한다”가 아니라,

> “나는 왜 계속 **기본기를 미루고** 있을까?”

에 가깝다.

## 5. 그래도 이제는 조금 다르게 해보려는 다짐

후회만 적어두고 끝내면 너무 우울하니까,  
이 글을 쓰면서 최소한 나 자신과 이런 약속을 해보기로 했다.

### 5.1 git: **진짜로 이해하고 넘어갈 리스트**

앞으로는 아래 네 가지를 **텍스트/그림으로 직접 정리**할 것이다.

1. **브랜치 모델**
   - `main`, `feature/*`, `hotfix/*` 같은 구조
   - 로컬 vs 원격 브랜치 관계
2. **주요 명령의 역할**
   - `commit`: 현재 변경을 스냅샷으로 남긴다
   - `merge`: 두 갈래의 히스토리를 합친다
   - `rebase`: 내 히스토리를 다른 베이스 위로 다시 쌓는다
   - `reset`: 브랜치 포인터를 “이전 시점”으로 되돌린다
3. **위험한 상황에서 쓸 수 있는 안전장치**
   - `git status`, `git log`, `git reflog`는 항상 먼저 확인하기
4. **“AI에게 묻기 전에 스스로 해보기” 룰**
   - 최소한 내가 하려는 명령을 **내 말로 한 줄 설명**해보고,
   - 그 설명이 말이 되는지 먼저 체크한 다음에 실행하기

AI는 여전히 사용할 것이다.  
하지만 **“내 생각을 검증하는 도구”**로 쓰고 싶다.  
“그냥 단계를 알려주는 리모컨”이 아니라.

### 5.2 GitHub 비밀번호/SSH 설정: 한 번은 내 손으로 문서를 만든다

이제는 다음을 내 레포의 `docs/` 폴더 어딘가에 남길 생각이다.

- EC2에서 GitHub에 **SSH 키로 접속하는 전체 절차**
  1. `ssh-keygen`으로 키 생성
  2. `~/.ssh/id_ed25519.pub`를 GitHub에 등록
  3. `git remote set-url origin git@github.com:...`으로 변경
- 필요 시 `credential.helper`로 토큰 캐싱하는 방법

이걸 AI에게 “복붙 받는 것”이 아니라,  
**내가 직접 적은 나만의 가이드**로 남겨 두고 싶다.

그래야 다음에 새 EC2를 만들 때  
이 글을 읽으면서 스스로 설정해 볼 수 있고,  
또 그 과정이 반복되면서 진짜 내 것이 될 테니까.

### 5.3 EC2 보안: 최소한 이 세 가지는 몸에 익히기

1. **보안 그룹**
   - 어떤 포트를 세상에 열지, 어떤 IP로 제한할지
2. **UFW (서버 내부 방화벽)**
   - SSH를 건드릴 땐 항상 세션 두 개 띄우기
3. **SSM (Session Manager)**
   - SSH가 막혀도 들어갈 수 있는 두 번째 통로 확보

이 세 가지는 “문제가 생기면 찾아보는 것”이 아니라,  
**문제가 생기기 전에 이해하고 세팅해 두는 것**이라고 받아들이려고 한다.

## 6. 마치며 – AI 시대에도, 기본기는 결국 나에게 남는다

AI 덕분에 개발이 편해진 건 사실이다.  
git이나 EC2 보안 설정 같은 것도 예전보다 훨씬 빠르게, 훨씬 덜 고통스럽게 처리할 수 있다.

하지만 이 글을 쓰면서 느낀 건 하나다.

> **문제를 대신 해결해 줄 수는 있어도,  
> 그 경험에서 배울지 말지는 여전히 내 선택이다.**

- git을 “영원히 이해 안 되는 도구”로 남길 수도 있고,
- 이번 기회에 “한 번쯤은 제대로 정리해 본 적 있는 도구”로 만들 수도 있다.
- EC2 보안을 “매번 검색해서 세팅하는 귀찮은 것”으로 볼 수도 있고,
- “내 서비스가 **진짜 서비스**가 되는 출발점”으로 볼 수도 있다.

나는 그동안 **편리함에 너무 기대고**,  
공부해야 할 타이밍을 몇 번씩 지나쳐 왔다.

그래서 이 글은  
“git 때문에 고생한 개발 기록”이라기보다,

> “AI 시대에도 결국 기본기를 피할 수는 없다는 걸  
>  이제야 조금씩 인정하는 개발자의 자기 고백”

에 가깝다.

언젠가 이 글을 다시 읽을 때쯤에는  
적어도 git 브랜치 그림 정도는 머릿속에서 그릴 수 있게 되어 있기를,  
그리고 EC2에서 더 이상 GitHub 비밀번호를 반복해서 치고 있지 않기를,  
조용히 바라본다.
